Neutrino is a **proof-aware** chain. Every block's state transition must be
covered by a zero-knowledge proof; chunk proofs aggregate block proofs;
recursive checkpoint proofs fold finalized chunks into a single succinct
attestation of all history since genesis.

This doc defines:

- The three-tier proof hierarchy.
- The block proof lifecycle.
- Chunk aggregation rules.
- The recursive checkpoint construction.
- The prover-role decomposition and economics.
- The proof-system abstraction (so we can swap concrete backends).

It depends on [03-execution-runtime](03-execution-runtime.md) for the VM,
[07-block-format](07-block-format.md) for the structures, and
[02-consensus](02-consensus.md) for the finality semantics.

## Why proof-aware

Without proofs, every node must re-execute every block. The chain can prune
old block bodies (saving disk) but cannot prune old execution traces
(needed for sync). A new node always has to either trust a recent checkpoint
*by reputation* or replay history.

With proofs:

- **New nodes verify**, they don't re-execute. Time to sync is `O(1)` in the
  number of historical blocks.
- **Light clients are first-class**. A 100 KB recursive proof + 32-byte state
  root suffices to verify the current state.
- **History pruning is mechanically safe**. Once a checkpoint is covered by a
  recursive proof, no honest node ever needs the underlying blocks to
  validate anything.

The cost is real: prover hardware, proof latency, and a more complex
consensus pipeline. The Neutrino design tolerates that cost in v1.

## Three-tier proof hierarchy

```
Block_h  ──proves──►  state_root_{h-1} → state_root_h
        │
        │  aggregated by chunk
        ▼
Chunk_c  ──proves──►  state_root_at_start_of_c → state_root_at_end_of_c
        │             + all block-level VRF / sig checks for blocks in c
        │             + block-hash continuity
        │
        │  folded recursively at finality
        ▼
RecursiveCheckpoint_n  ──proves──►  genesis_state_root → checkpoint_state_root_n
                                   + every chunk in [1..n] was BFT-finalized
                                   by 2/3 of its epoch's validator set
```

Each tier is a separate proof artifact and a separate market.

## Block proof

### What it proves

```
public inputs:
    chain_id
    height
    parent_block_hash
    block_hash
    state_root_before
    state_root_after
    transactions_root
    receipt_root  (optional; runtime-defined)
    da_root
    vm_code_hash       // which runtime ELF was authoritative at this height
    abi_version

statement:
    There exist transactions T and a state witness W (trie nodes) such that
        Hash(T) = transactions_root,
        VmExecute(vm_code_hash, T, state_root_before, W) =
            (state_root_after, receipt_root),
        Hash(Body) = da_root, where Body is the proposer-published block body.
    The witness W is generated by execution, not published by the proposer;
    it is fetched separately by the prover from a witness-serving peer.
```

The witness `W` includes the trie nodes the execution touched (Merkle
witnesses for state reads), but never appears in the public inputs.

### Where the proof lives

A block proof is **not** part of the gossiped block. Including it would:

- couple proposer hardware requirements to prover hardware,
- inflate block size unnecessarily,
- prevent the prover market.

Instead, the proof is a separate object referenced by the block via the
`block_hash` it proves. Provers publish proofs on a dedicated gossip topic
(see [06-networking](06-networking.md)).

### Witness format

The block prover does not re-fetch state. It consumes a `SealedWitness`
the runtime host emits during execution. The witness is borsh-encoded,
persisted in the `witnesses` storage column under the block hash, and
shipped peer-to-peer via `/neutrino/req/witness_by_block/1`.

```rust
pub struct SealedWitness {
    pub parent_state_root:     [u8; 32],
    pub block_context:         BlockContextWitness,
    pub runtime_input:         Vec<u8>,   // bytes served by host_input
    pub block_header:          Vec<u8>,   // borsh(Header)
    pub block_body:            Vec<u8>,   // borsh(Body)
    pub state_reads:           Vec<StateRead>,
    pub state_next_key_reads:  Vec<StateNextKeyRead>,
}

pub struct StateRead {
    pub key:        Vec<u8>,
    pub base_value: Option<Vec<u8>>,        // None for exclusion
    pub proof:      neutrino_trie::Proof,   // anchored at parent_state_root
}

pub struct StateNextKeyRead {
    pub prefix:            Vec<u8>,
    pub after:             Vec<u8>,
    pub result_key:        Option<Vec<u8>>,
    pub result_base_value: Option<Vec<u8>>,
    pub result_proof:      Option<neutrino_trie::Proof>,
}
```

The proof system anchors every `StateRead.proof` against
`SealedWitness.parent_state_root`. `StateNextKeyRead` records cursor
reads separately so syscall replay can constrain key iteration results
instead of silently omitting them from the private input. Writes are not
recorded explicitly: the prover replays the trace to derive post-state
values, then checks the post-state root equals the header's
`state_root`. Reads that the runtime served from its dirty overlay (key
written and re-read in the same block) still carry a base-trie proof so
the prover never has to trust the runtime's overlay state; the live
overlay value is reconstructed from the trace.

### Block prover backend

The v1 block backend is a **custom Plonky3 STARK** that proves
correct execution of the canonical RV32IM ELF identified by
`vm_code_hash`. The block prover is therefore an in-tree zkVM, not a
wrapper around an external backend (SP1, RISC Zero, Jolt). This
trade-off is detailed under [Alternatives considered](#alternatives-considered).

- **Field.** BabyBear, `p = 2^31 - 2^27 + 1`. Native 31-bit arithmetic
  matches the dominant Plonky3 deployments, has the best FRI tooling,
  and decomposes RV32IM's 32-bit operands with one extra bit-witness
  per word.
- **Hash.** Poseidon2 over BabyBear. Arithmetic-friendly hash with
  practical in-circuit Merkle and Fiat-Shamir performance; mandatory if
  the chunk (M9) and recursive (M10) layers are to verify block proofs
  cheaply in-circuit. The chain's consensus hash remains BLAKE3
  (`primitives::blake3_256`); the two hashes occupy disjoint namespaces.
- **Recursion target.** Block proofs are the leaves of the Plonky3
  recursion tree built by M9 (chunk) and M10 (recursive checkpoint).
  Keeping all three tiers on Plonky3 / BabyBear / Poseidon2 removes one
  cross-backend wrapping step from the chunk circuit.

#### AIR decomposition

The block prover is not a single STARK. It is a set of AIRs sharing a
logUp lookup bus, each constraining one shape of work:

| AIR | Trace shape | Role |
|---|---|---|
| Range check (u8 / u16 / u32) | Static lookup tables | Source of truth for byte and word range arguments every other AIR consumes. |
| Program ROM | One row per `(pc, instruction)` pair in the ELF | Anchors PC → instruction lookups to `vm_code_hash`. |
| Memory consistency | One row per memory access | Sorted multi-set over `(addr, ts, op, val)`; checks reads return latest writes; ingests the witness's `state_reads` as the boundary for trie-backed memory. |
| Base RV32I CPU | One row per executed RV32I instruction | Register file, ALU, branches, jumps, loads/stores via the memory bus. |
| M-extension | One row per multiply/divide | MUL/MULH/MULHU/MULHSU/DIV/DIVU/REM/REMU with the M0 non-trapping semantics for `/0` and signed overflow. |
| Traps + system | One row per ECALL/EBREAK/illegal/alignment/OOM | Encodes the [`Trap`](../../crates/vm-rv32im/src/lib.rs) variants and gas accounting. |
| Syscall replay | One row per `ECALL` | Binds the syscall number, request, and response bytes to the host ABI; gas-consumed constraint matches the per-syscall cost table from [04-host-abi](04-host-abi.md). |

Cross-AIR consistency lives on the lookup bus: each AIR sends/receives
typed records (memory accesses, byte-range queries, program-rom rows,
syscall pairs) and the multi-set permutation closes globally per
shard. No AIR can fabricate a memory access or a syscall response
without a matching counterpart on another AIR.

#### Continuations

A block whose RV32IM trace exceeds `TRACE_LIMIT` rows splits into
**shards**. Each shard is a full multi-AIR STARK; consecutive shards
are linked by an argument committing to the boundary state: register
file, program counter, last memory commitment, and gas remaining. The
final shard's boundary state must match the post-execution
commitments encoded in the public inputs. `TRACE_LIMIT` is a backend
constant tuned against measured prover memory and FRI cost; the
public-input surface is identical regardless of shard count.

#### Public-input commitment

The borsh-encoded `BlockProofPublicInputs` is hashed under Poseidon2
to an eight-BabyBear-element digest committed as the STARK's public
values. The verifier recomputes the digest from the same
fields and rejects mismatches. The mock backend (M2 through M7) uses
BLAKE3 under `MOCK_BLOCK_DOMAIN`; the real backend's Poseidon2 digest
is a separate namespace.

### Proof deadline

```
proof_deadline = block_slot + PROOF_WINDOW
PROOF_WINDOW = 8 slots  (default; ~32 s; tunable)
```

The window is short enough that finality is not blocked for long, and long
enough that the prover market can fulfill it for typical blocks. If proofs
systematically lag, chunks accumulate and finality stalls — the chain
preserves safety over liveness. Adaptive chunking is a v1.1 lever.

## Chunk proof

### Chunk definition

```rust
pub const CHUNK_SIZE: u32 = 128;

pub struct Chunk {
    pub chunk_id: u64,                  // floor((start_height - 1) / CHUNK_SIZE)
    pub start_height: u64,
    pub end_height: u64,                 // inclusive
    pub start_state_root: [u8; 32],
    pub end_state_root: [u8; 32],
    pub block_hash_root: [u8; 32],       // Merkle root over included block hashes
    pub block_proof_root: [u8; 32],      // Merkle root over included block-proof commitments
    pub vrf_proof_root: [u8; 32],        // Merkle root over included VRF proofs
    pub active_validator_set_root: [u8; 32], // active set used for this chunk
    pub next_validator_set_root: [u8; 32],   // post-chunk set for next chunk
    pub da_root: [u8; 32],
}
```

A chunk **only** becomes eligible for the BFT vote once every block in its
range is `Proven`.

### What a chunk proof proves

```
public inputs:
    chain_id
    chunk_id
    start_height, end_height
    start_state_root, end_state_root
    block_hash_root
    vrf_proof_root
    active_validator_set_root
    next_validator_set_root
    da_root
    vm_code_hash_root      // covers any runtime upgrade inside the chunk
    abi_version

    statement:
    block_{start_height}.parent_hash = CP_{n-1}.end_block_hash.
    state_root_before_{start_height} = start_state_root.
    For each height h in [start_height ..= end_height]:
        BlockProof_h is valid.
        If h > start_height:
            block_hash_h.parent = block_hash_{h-1}.
            state_root_after_{h-1} = state_root_before_h.
        proposer of block_h was VRF-eligible at slot_h under
            active_validator_set_root and seed_from_prior_checkpoint.
        block_h signature by proposer is valid.
    state_root_after_{end_height} = end_state_root.
    active_validator_set_root = CP_{n-1}.end_validator_set_root.
    next_validator_set_root is derived from runtime state at end_state_root.
```

The chunk proof composes block proofs. v1 uses **recursive verification**
inside the same Plonky3 / BabyBear / Poseidon2 stack the block prover
emits: the chunk circuit re-runs the block-proof FRI verifier
in-circuit and accumulates across blocks. This is the highest per-block
cost option but the simplest in terms of semantics, and the
cross-backend wrapping overhead that motivates folding schemes does
not exist because every proof in the recursion shares one backend.

Alternative aggregation strategies the `ProofSystem` trait still
permits:

1. **Folding scheme.** Nova / SuperNova / HyperNova fold block
   instances into one with an external "decider" step. Lower per-block
   overhead, more moving pieces.
2. **Dedicated chunk circuit.** Aggregate via Plonkish lookup
   arguments without explicitly verifying each block proof.
   Lowest cost, tightest coupling to the in-tree backend.

Either can become the v1.x default once measured against the M9
baseline; the `proof_system_version` field on the recursive checkpoint
public inputs is the upgrade lever.

### Adaptive chunking

Default chunk size is 128 canonical blocks (roughly 8.5 min at 4 s slots if no
slots are empty). v1.1 may add two escape hatches:

- **Timeout chunk.** If a chunk's proof is not ready by `chunk_timeout`,
  emit an undersized chunk for whatever was proven, and start a fresh
  chunk for the remainder.
- **Emergency chunk.** If finality is stalling for more than
  `FINALITY_STALL_THRESHOLD`, the protocol allows a chunk as small as 1 block.
  Pure liveness lever; expensive in per-block proof overhead.

Adaptive chunking is a v1.1 addition. v1 ships fixed-size chunks.

## Finality vote on a chunk

When a chunk is `ChunkProven`, every active validator votes on it using the
round/lock rules from doc 02. Two phases, structured like Tendermint:

```
prevote   →  prevote BLS aggregate, 2/3 weight for (chunk_id, round, chunk_hash)
precommit →  precommit BLS aggregate, 2/3 weight for the same tuple
```

See [02-consensus](02-consensus.md) for the full finality gadget. The
finality certificate contains both the prevote and precommit aggregates plus
their signer bitmaps.

## Recursive checkpoint proof

### What it proves

For checkpoint `n` (one per finalized chunk):

```
public inputs:
    chain_id
    genesis_hash
    checkpoint_id            == n
    checkpoint_height        == Chunk_n.end_height
    checkpoint_block_hash    == hash of last block in Chunk_n
    checkpoint_state_root    == Chunk_n.end_state_root
    active_validator_set_root == active set used for FinalityCert_n
    next_validator_set_root   == state-derived active set for Chunk_{n+1}
    previous_checkpoint_hash == hash of CP_{n-1}
    proof_system_version
    vm_code_hash             // current authoritative runtime
    abi_version

statement (informal):
    Either n = 1 and the chain begins from genesis with state_root = genesis_state_root,
    Or there exists a previous recursive proof Π_{n-1} for CP_{n-1} such that
        Verify(Π_{n-1}, CP_{n-1}_public_inputs) = true,
        Verify(ChunkProof_n, Chunk_n_public_inputs) = true,
        Verify(FinalityCert_n, FinalityCert_n_message,
            active_validator_set_n) = true,
        active_validator_set_n = CP_{n-1}.end_validator_set_root,
        next_validator_set_n derives correctly from Chunk_n.end_state_root
            via the runtime validator-set transition rules,
        Chunk_n.start_state_root = CP_{n-1}.checkpoint_state_root,
        Chunk_n.end_state_root   = CP_n.checkpoint_state_root,
        CP_n.end_validator_set_root = next_validator_set_n.
```

### Recursive transition

```rust
pub fn recursive_step(
    prev_cp_public:   &CheckpointPublicInputs,
    prev_cp_proof:    &RecursiveProof,
    chunk_public:     &ChunkPublicInputs,
    chunk_proof:      &ChunkProof,
    finality_cert:    &FinalityCert,
    validator_set_witness: &ValidatorSetTransitionWitness,
) -> Result<(CheckpointPublicInputs, RecursiveProof), ProofError>
```

The new recursive proof verifies all three sub-proofs **in-circuit** so that
its public inputs are sufficient for a light client.

### Verifying the finality certificate in-circuit

Three options, in increasing efficiency:

1. **Full in-circuit BLS aggregate verification.** Add up all active
   pubkeys whose bit is set in the bitmap, hash-to-curve the message,
   compute the pairing, accept iff `e(agg_pk, H(m)) = e(g1, agg_sig)` and
   `signed_weight ≥ 2/3 * total_weight`. Heaviest but simplest.

2. **Two-step**: a separate **signature proof** attests to the finality
   cert's validity; the recursive checkpoint proof verifies the signature
   proof (one pairing or one SNARK verification) instead of the cert
   directly. Reduces recursion cost at the cost of one more proving step
   per chunk.

3. **Lookup-table BLS verification** in a Plonkish system. State of the art
   in 2024–25 (e.g. Eigen Labs, Polyhedra) brings BLS-381 pairing inside
   Halo2-style proofs to ~10⁵ constraints. We assume this regime.

We design for option 1 (simplest, no second-proof artifact) with option 2
as a fallback if the in-circuit BLS cost is unacceptable. The `ProofSystem`
trait abstracts this choice.

### Validator-set transitions inside the recursion

A checkpoint may include validator activations, exits, and slashings. The
recursive step verifies that:

```
validator_set_n = ApplyValidatorSetDiff(
    validator_set_{n-1},
    activations_in_chunk_n,
    exits_in_chunk_n,
    slashings_in_chunk_n,
)
```

The diff is computed by the runtime and committed in the chunk; the
recursive circuit re-applies it deterministically. The validator set is
represented as a Merkle tree of (pubkey, stake, status) records to keep
the in-circuit update small (log N per change).

## Proof system abstraction

```rust
pub trait ProofSystem: Send + Sync + 'static {
    type BlockProof:      Serialize + DeserializeOwned;
    type ChunkProof:      Serialize + DeserializeOwned;
    type RecursiveProof:  Serialize + DeserializeOwned;
    type SetupParams:     Sync;

    fn prove_block(
        &self,
        public: &BlockPublicInputs,
        witness: &BlockWitness,
    ) -> Result<Self::BlockProof>;

    fn verify_block(
        &self,
        public: &BlockPublicInputs,
        proof: &Self::BlockProof,
    ) -> Result<()>;

    fn aggregate_chunk(
        &self,
        public: &ChunkPublicInputs,
        block_proofs: &[Self::BlockProof],
    ) -> Result<Self::ChunkProof>;

    fn verify_chunk(
        &self,
        public: &ChunkPublicInputs,
        proof: &Self::ChunkProof,
    ) -> Result<()>;

    fn recurse(
        &self,
        prev: Option<(CheckpointPublicInputs, Self::RecursiveProof)>,
        chunk: (&ChunkPublicInputs, &Self::ChunkProof),
        cert: &FinalityCert,
        diff: &ValidatorSetTransitionWitness,
    ) -> Result<(CheckpointPublicInputs, Self::RecursiveProof)>;

    fn verify_recursive(
        &self,
        public: &CheckpointPublicInputs,
        proof: &Self::RecursiveProof,
    ) -> Result<()>;
}
```

Every component that touches proofs takes a `dyn ProofSystem`. Tests use a
mock backend that produces trivial proofs (just hashes).

### v1 stack

All three tiers ship on the same backend:

| Tier | Backend | Notes |
|---|---|---|
| Block | Custom Plonky3 STARK over BabyBear | In-tree zkVM. AIR decomposition above. Poseidon2 hash. |
| Chunk | Custom Plonky3 circuit | Aggregates 128 block proofs. Verifies their Plonky3 STARKs natively without a cross-backend wrapper. |
| Recursive checkpoint | Plonky3 → SNARK wrapper (Groth16 or PLONK over BN254) | Tiny final proof; light clients verify one Plonky3 → SNARK proof per checkpoint. |

The mock backend ([`MockProofSystem`](../../crates/proof-system/src/mock.rs))
remains the M2 through M7 stand-in. M8 onwards swaps in real backends.
`proof_system_version` in `RecursivePublicInputs` lets governance
upgrade the backend without invalidating old proofs (see
[Versioning](#versioning)).

### Alternatives considered

Three off-the-shelf RV32IM zkVMs were viable v1 block backends:

| Backend | Strengths | Why not v1 |
|---|---|---|
| **SP1** (Succinct) | Plonky3-based Rust zkVM; recursion-ready; production users. | Single-vendor coupling to the `succinct` toolchain (`sp1up`, `cargo-prove`). Block-proof outputs wrap inside Plonky3 for the chunk circuit at non-trivial cost. |
| **RISC Zero** | Mature, well-documented, Groth16 wrap built in. | Older STARK; larger block proofs; weaker recursion story. Cross-backend wrap into Plonky3 for chunk aggregation is expensive. |
| **Jolt** (a16z) | Lookup-heavy, fast prover. | Less mature; recursion path still developing; same cross-backend wrap problem as RISC Zero. |
| **Nexus** | Nova / SuperNova folding all the way up. | Earliest stage of the four; unclear long-term backend stability. |

We chose to build a custom Plonky3 RV32IM zkVM because

1. **Recursion uniformity.** All three tiers (block, chunk, recursive)
   run on Plonky3 / BabyBear / Poseidon2. The chunk circuit verifies
   block proofs natively rather than verifying a wrapped SP1 / RISC Zero
   proof inside Plonky3, which materially reduces chunk-circuit cost.
2. **No vendor lock-in.** The block prover lives in-tree under the
   workspace lints and license. There is no second toolchain to track
   for reproducible builds.
3. **Public-input control.** We bind exactly the
   [`BlockProofPublicInputs`](../../crates/consensus-types) shape; no
   adapter layer translates between an external zkVM's commitment and
   ours.

The cost is real and we own it. Writing a sound RV32IM AIR is large
engineering work: full opcode coverage, memory consistency, range
checks, lookup arguments, M-extension corner cases, continuations.
The audit surface is ours. The [Roadmap](09-roadmap.md) M8 entry
breaks the work into 14 sub-slices so progress is land-by-slice rather
than a single big-bang circuit.

A `proof_system_version = 2` upgrade can later switch to a different
block backend without disturbing the chunk and recursive tiers; the
recursion circuit verifies whichever block-proof shape matches its
own `proof_system_version`.

## Network roles

### Validator

Stakes, participates in finality voting. **May** produce blocks (if
VRF-eligible) and **may** run a prover, but neither is required. A validator
needs only enough resources to run the engine, verify proofs, and sign votes.

### Block producer

A validator that won a VRF slot. Publishes the block. Responsible for
ensuring the block becomes `Proven` by the deadline, either by running a
prover or outsourcing to the prover market.

### Block prover

A node (validator or not) that consumes pending blocks, generates block
proofs, and gossips them. Compensated per accepted proof.

### Chunk aggregator

A node that produces chunk proofs from N block proofs. Compensated per
accepted chunk proof.

### Checkpoint prover

A node that runs the recursive folding step. Exactly one recursive proof
per finalized chunk. Compensated per accepted recursive proof.

### Fallback prover

A block prover that steps in after a block's proof deadline expires. Earns
a bounty paid from the original proposer's deposit.

These roles are **separable**: any node may run any subset. Operators with
specialized GPU/FPGA hardware likely concentrate on proving; operators
with home connections likely concentrate on validation.

## Prover economics

### Rewards (per finalized chunk; paid by runtime, scheduled by engine)

```
R_block       = base_block_reward_per_block
R_proof       = prover_reward_per_block_proof
R_chunk_agg   = chunk_aggregator_reward
R_checkpoint  = recursive_checkpoint_reward
R_finality    = finality_vote_reward_per_validator
R_fallback    = fallback_proof_bounty   // strictly > R_proof
```

The runtime is responsible for the exact reward schedule and split. The
engine surfaces who-did-what in the block outcome.

### Penalties

```
P_missed_proof        = proposer or delegated primary prover fails to deliver
                       proof in window → loses proof reward; fallback bounty
                       can be paid from proposer/prover bond depending on
                       delegation contract
P_invalid_block       = proposer produced unprovable transition
                       → slashed
P_invalid_proof_spam  = prover submits invalid proof
                       → bond burned, reputation slashed
P_equivocation        = double propose / conflicting finality vote
                       → slashed (see 02-consensus.md)
P_da_commitment_fraud = proposer published bytes that do not open/hash to
                       da_root → slashed
P_unavailable_block   = body/witness cannot be fetched in v1 full-gossip DA
                       → block excluded / rewards forfeited, not slashable
```

### Designing the fallback bounty

Goal: fallback proving is profitable enough to **guarantee liveness**, but
not so profitable that proposers intentionally skip and let the fallback
prover collect.

A simple working scheme:

```
R_fallback       = R_proof * (1 + urgency)
urgency          = (slots_past_deadline / SLOT_BUDGET_PER_CHUNK) * fallback_premium
SLOT_BUDGET_PER_CHUNK = CHUNK_SIZE   // upper bound on slots a chunk consumes
                                     // when every slot is filled; "ideal" length
fallback_premium = 0.5               // tunable
```

`SLOT_BUDGET_PER_CHUNK` is the *ideal* (no-empty-slot) slot count for a chunk.
Real chunks may take longer if slots are empty; using the ideal as the
denominator makes urgency monotonic and predictable regardless of empty-slot
rate.

The fallback bounty grows with urgency, but the proposer's loss
(`P_missed_proof + R_block forfeited if block excluded`) dominates the
benefit of skipping by design.

## Liveness vs safety under prover stress

The chain prioritizes **safety** over **liveness**: a chunk cannot finalize
without a valid proof, full stop. Under prover stress (slow hardware, big
state diffs, congestion):

- Pending chunks accumulate.
- Fixed-size chunks accumulate behind the proof bottleneck; adaptive chunking
  is explicitly deferred to v1.1.
- Finality halts if no chunk can be proven at all.
- Block production continues (proposers VRF-elect and publish blocks); the
  unfinalized tail grows.

The chain never finalizes incorrect state, but it can lag arbitrarily far
behind under prover failure. The economic incentives and the open prover market
are the v1 liveness levers; adaptive chunking is a post-v1 lever.

## Failure modes

### Prover bottleneck

Block proofs slow. Mitigations: fallback prover market, proof-window tuning,
specialized prover hardware, future faster backends. Adaptive chunks are v1.1.

### Invalid block built upon

Detected when the block proof fails. Fork choice immediately disregards the
unprovable branch and re-orgs to the tip of the proven branch. Engine
slashes the proposer.

### Finality without proof

Forbidden by protocol. A finality cert over an unproven chunk is invalid;
the recursive circuit cannot verify it; the cert is rejected by every honest
node. If 2/3 of validators sign one anyway, that's a 1/3-quorum slashing
condition.

### Unavailable block data

Proposer publishes a header but the body or witness bundle is unavailable, so
provers cannot generate the block proof. In v1 this is not objectively
slashable by itself; absence is a network observation, not consensus evidence.
The block is excluded once proof deadlines expire and rewards are forfeited.
If a proposer publishes bytes that fail to open to `da_root`, that is objective
`DaCommitmentFraud` and is slashable.

## Data availability

Proof generation requires access to:

- The block's transactions.
- The trie nodes the execution read (Merkle witnesses).
- Any runtime-specific witness data (e.g. signatures over the body).

The block commits to a `da_root` covering the proposer-published body. The
execution witness (trie nodes) is generated by the host and stored separately
in the `witnesses` column; provers fetch it via `witness_by_block`. A proposer
that fails to make the body available loses rewards and risks block exclusion;
slashing requires an objective bad opening or mismatched bytes.

Higher-throughput DA modes (erasure-coded sampling, DA committees, external
DA layers) are post-v1 and can be added behind the `da_root` field without
breaking the proof structure.

## Versioning

The proof system version is part of the recursive proof's public inputs.
This lets the chain upgrade proof systems via governance without
invalidating old proofs:

- New proofs use the new system.
- A migration step proves "old recursive proof at CP_n is valid" inside the
  new system, producing a fresh recursive proof in the new system at CP_n.
- Light clients verify whichever proof system matches the proof they
  downloaded.

ABI version (`abi_version`) is committed inside every proof so a runtime
upgrade that changes the host syscall table forces a new proof system
configuration, not a silent shift.
