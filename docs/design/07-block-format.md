# 07 — Block, Chunk, Checkpoint, and Proof Formats

This file defines every authenticated wire shape: `Header`, `Body`, `Block`,
`Chunk`, `FinalityVote`, `FinalityCert`, `Checkpoint`, and the three proof
artifacts (`BlockProof`, `ChunkProof`, `RecursiveCheckpointProof`).

The engine *only* deeply understands the `Header` and the proof artifacts. The
body is opaque to it — the engine builds roots and hands the body to the
runtime to execute.

---

## 7.1 Encoding

We use **SCALE** (Simple Concatenated Aggregate Little-Endian) via the
`parity-scale-codec` crate.

Why SCALE:

- Compact, deterministic, well-specified.
- Battle-tested in Polkadot/Substrate.
- Rust derive macros make field-by-field encoding trivial and refactor-safe.
- No reflection — types are encoded structurally; field order is part of the
  schema.

Why not SSZ:

- SSZ shines for in-place merkle proofs over consensus structures. Our
  Merkle-proof needs (state and history inclusion) are served by the binary
  sparse Merkle trie in [05-state-and-storage](05-state-and-storage.md).
- We may add SSZ for light-client friendliness later. The `Codec` trait in the
  `codec` crate isolates the choice (see [08-crate-layout](08-crate-layout.md)).

## 7.2 Header (engine-canonical schema)

The header is the **only** structure the engine deeply understands. The body
is opaque.

```rust
pub struct Header {
    /// Protocol version. Bumped on consensus-breaking changes.
    pub version:            u32,

    /// Monotonically increasing block number since genesis.
    /// Distinct from `slot` because some slots are empty (no VRF winner).
    pub height:             u64,

    /// Slot at which this block was produced.
    pub slot:               u64,

    /// Hash of the parent header.
    pub parent_hash:        [u8; 32],

    /// Validator index of the proposer, into the active validator set for the
    /// parent checkpoint / current chunk.
    pub proposer_index:     u32,

    /// BLS signature over
    /// `"NEUTRINO_VRF" || chain_id || finalized_seed || slot`.
    /// Doubles as a verifiable random function output: the SHA-256 hash of
    /// this field is the VRF output, used to check the stake-weighted
    /// proposer threshold. See 12-randomness.md.
    pub vrf_proof:          [u8; 96],   // BLS G2 sig, compressed

    /// Root of the runtime-controlled state AFTER this block executes.
    pub state_root:         [u8; 32],

    /// Root over the canonical list of transactions in the body.
    pub transactions_root:  [u8; 32],

    /// Root over the finality votes included in the body.
    /// Finality votes are prevote/precommit messages on past chunks.
    pub votes_root:         [u8; 32],

    /// Root over slashing-evidence items in the body.
    pub slashings_root:     [u8; 32],

    /// Root over deposits + voluntary_exits items in the body.
    pub validator_ops_root: [u8; 32],

    /// Commitment to data-availability information for this block.
    /// In v1 this is the binary merkle root over the full block body
    /// (transactions, finality votes, slashings, deposits, voluntary exits)
    /// using BLAKE3. The execution witness (state trie nodes) is NOT
    /// covered — it is generated by the host after execution and stored
    /// separately. Future erasure-coded DA schemes use this field without
    /// protocol break, attaching the extended bundle's root here.
    pub da_root:            [u8; 32],

    /// 32-byte runtime-defined extra. Lets the runtime commit to additional
    /// data without forcing engine schema changes. Engine ignores contents.
    pub runtime_extra:      [u8; 32],

    /// Gas used by this block.
    pub gas_used:           u64,

    /// Gas limit set by the proposer (clamped by the protocol-level cap).
    pub gas_limit:          u64,

    /// Wall-clock timestamp at the start of the slot (seconds since UNIX
    /// epoch). Engine bounds vs. local clock to reject far-future blocks.
    pub timestamp:          u64,

    /// BLS signature by the proposer over the SCALE-encoded header excluding
    /// this field.
    pub signature:          [u8; 96],   // BLS G2 sig, compressed
}
```

Header changes from v1 (original Gasper draft):

- **Added** `height: u64` — explicit block number distinct from slot.
- **Renamed** `randao_reveal` → `vrf_proof`. Semantics now: VRF proof for
  *leader-election*, no longer a randomness contribution per se. The mix is
  derived post-finalization (see [12-randomness](12-randomness.md)).
- **Renamed** `attestations_root` → `votes_root`. Votes are *chunk-level*
  prevotes/precommits, not slot-level attestations.
- **Added** `validator_ops_root` — replaces implicit hashing of deposits/exits.
- **Added** `da_root` — explicit DA commitment, distinct from transactions
  root, so future DA schemes (erasure coding, sampling) can plug in.
- **Kept** `runtime_extra` — runtime escape hatch.

The header hash is `H(SCALE(header_without_signature))` where `H = BLAKE3`.
The signature binds the proposer's BLS key to that hash.

## 7.3 Body (engine-opaque, runtime-interpretable)

```rust
pub struct Body {
    /// Each transaction is a runtime-defined blob.
    pub transactions:     Vec<Bytes>,

    /// Finality votes on past chunks. Aggregated BLS sigs.
    pub finality_votes:   Vec<FinalityVote>,

    /// Slashing evidence detected by the proposer.
    pub slashings:        Vec<SlashingEvidence>,

    /// Validator activation/exit ops.
    pub deposits:         Vec<Deposit>,
    pub voluntary_exits:  Vec<VoluntaryExit>,
}
```

The engine builds the five roots from these vectors and verifies them against
the header. Mismatch → block invalid, dropped from fork choice.

## 7.4 Finality vote (chunk-level)

```rust
pub enum FinalityVotePhase {
    Prevote,
    Precommit,
}

pub struct FinalityVoteData {
    /// Chunk being voted on.
    pub chunk_id:    u64,
    /// Tendermint-style round for this chunk.
    pub round:       u32,
    /// Hash of the chunk being voted on.
    pub chunk_hash:  [u8; 32],
    /// Phase of the vote.
    pub phase:       FinalityVotePhase,
}

pub struct FinalityVote {
    /// Bitmap of which validators in the active set signed.
    pub aggregation_bits: BitVec,
    pub data:             FinalityVoteData,
    /// Aggregate BLS signature (G2) over the SCALE-encoded `data` with
    /// the per-phase domain tag.
    pub signature:        [u8; 96],
}
```

Domain tags for signing:
- Prevote: `"NEUTRINO_PREVOTE" || chain_id || chunk_id || round`
- Precommit: `"NEUTRINO_PRECOMMIT" || chain_id || chunk_id || round`

This prevents cross-phase replay.

A block proposer may include multiple votes per chunk/round (different
aggregator groups across the validator set). The consensus engine is
responsible for deduplicating and combining them into one `FinalityCert` per
finalized chunk. The runtime only sees resulting reward/slashing inputs.

## 7.5 Slashing evidence

Eight objective slashing conditions are defined in [02-consensus §slashing](02-consensus.md).
We encode them as a single enum, growing as conditions are added.

```rust
pub enum SlashingEvidence {
    /// Two distinct headers signed by the same proposer at the same slot.
    DoubleProposal {
        proposer_index: u32,
        header_a:       SignedHeader,
        header_b:       SignedHeader,
    },

    /// Signed header claimed eligibility with an invalid/wrong-seed/wrong-slot
    /// VRF proof or with an output above the stake-weighted threshold.
    InvalidVrfClaim {
        proposer_index: u32,
        header:         SignedHeader,
        reason:         VrfRejectionReason,
    },

    /// Two distinct prevotes by the same validator on the same
    /// (chunk_id, round) with different chunk_hash.
    DoublePrevote {
        validator_index: u32,
        vote_a:          IndexedVote,
        vote_b:          IndexedVote,
    },

    /// Same, precommit phase.
    DoublePrecommit {
        validator_index: u32,
        vote_a:          IndexedVote,
        vote_b:          IndexedVote,
    },

    /// Tendermint lock violation: a validator precommits a conflicting chunk
    /// without valid higher-round unlock evidence.
    LockViolation {
        validator_index: u32,
        vote_a:          IndexedVote,
        vote_b:          IndexedVote,
        lock_evidence:   LockEvidence,
    },

    /// Validator signed a chunk whose proof later turned out Invalid.
    InvalidProofSigning {
        validator_index: u32,
        vote:            IndexedVote,
        invalid_proof_evidence: BlockProofRejection,
    },

    /// Validator signed in a fork that diverges from a finalized chunk.
    LongRangeForkParticipation {
        validator_index: u32,
        vote:            IndexedVote,
        canonical_finalized_chunk: Checkpoint,
    },

    /// Proposer signed a header whose published DA bundle is inconsistent with
    /// da_root. Pure unavailability is not objectively slashable in v1.
    DaCommitmentFraud {
        proposer_index: u32,
        header:         SignedHeader,
        fraud_proof:    DaFraudProof,
    },
}

pub struct IndexedVote {
    pub data:      FinalityVoteData,
    pub signature: [u8; 96],
}

pub enum VrfRejectionReason {
    BadSignature,
    WrongDomain,
    WrongFinalizedSeed,
    WrongSlot,
    ThresholdNotMet,
}

pub struct BlockProofRejection {
    pub block_hash: [u8; 32],
    pub proof_hash: [u8; 32],
    pub verifier_version: u32,
    pub reason: ProofRejectionReason,
}

pub enum ProofRejectionReason {
    MalformedProof,
    PublicInputsMismatch,
    VerifierRejected,
}

pub struct LockEvidence {
    /// The earlier lock certificate the validator was bound by.
    pub locked_prevote_quorum: QuorumCertificate,
    /// Optional higher-round quorum that would have unlocked it. Missing or
    /// invalid unlock evidence makes the later precommit slashable.
    pub claimed_unlock_quorum: Option<QuorumCertificate>,
}

pub struct DaFraudProof {
    pub expected_da_root: [u8; 32],
    pub computed_da_root: [u8; 32],
    pub bundle_hash: [u8; 32],
    pub offending_bundle: Bytes,
}
```

`SignedHeader` is the normal signed header wrapper. Rejection/fraud helper
types are encoded here so slashing evidence is self-contained; the verifier
logic behind them is specified in [10-proof-system](10-proof-system.md) and the
consensus/networking crates.

## 7.6 Deposit / VoluntaryExit

```rust
pub struct Deposit {
    pub pubkey:                 [u8; 48],   // G1 compressed
    pub withdrawal_credentials: [u8; 32],
    pub amount:                 u64,
    pub signature:              [u8; 96],   // BLS proof-of-possession
}

pub struct VoluntaryExit {
    pub validator_index: u32,
    pub epoch:           u64,
    pub signature:       [u8; 96],
}
```

The engine surfaces these to the runtime via the block body; the runtime
applies activation/exit semantics and updates the validator-set entry in
state.

## 7.7 Transactions

A transaction is `Bytes` to the engine. The runtime defines:

- the signature scheme,
- the nonce model,
- the fee model,
- the maximum size.

The engine enforces only:

- per-transaction max byte size from `ChainSpec`,
- block-level total byte size,
- block-level gas limit.

## 7.8 Chunk

A chunk groups consecutive *proven* blocks for finality and recursive proof.

```rust
pub struct Chunk {
    /// Sequential chunk number, starting at 0.
    pub chunk_id:           u64,

    /// First and last block heights in this chunk (inclusive).
    pub start_height:       u64,
    pub end_height:         u64,

    /// State root at the START of the chunk (before first block).
    pub start_state_root:   [u8; 32],
    /// State root at the END of the chunk (after last block).
    pub end_state_root:     [u8; 32],

    /// First and last block hashes (for cheap chain continuity check).
    pub start_block_hash:   [u8; 32],
    pub end_block_hash:     [u8; 32],

    /// Merkle root of the chunk's block hashes (binary tree over heights).
    pub block_hash_root:    [u8; 32],

    /// Merkle root of the chunk's block-proof hashes (binary tree).
    pub block_proof_root:   [u8; 32],

    /// Merkle root of the VRF proofs (binary tree over slots in range).
    pub vrf_proof_root:     [u8; 32],

    /// Active set used for every proposer check and finality vote in this
    /// chunk. Must equal the previous checkpoint's end_validator_set_root.
    pub active_validator_set_root: [u8; 32],

    /// State-derived validator set after applying the whole chunk. This becomes
    /// the active set for the next chunk once this chunk checkpoints.
    pub next_validator_set_root:   [u8; 32],

    /// DA commitment root over all blocks' da_root values.
    pub da_root:            [u8; 32],
}
```

`chunk_id` is monotonically increasing and gap-free. Chunk 0 covers canonical
heights `1..=CHUNK_SIZE`; checkpoint 0 is genesis. Empty slots create gaps in
slot numbers, not block heights. Multi-winner slots create competing branches;
only the canonical branch contributes a block at a height.

The chunk's hash is `H(SCALE(chunk))`.

## 7.9 Finality certificate

A finality certificate aggregates votes on a chunk and proves it crossed the
2/3 threshold in both phases.

```rust
pub struct FinalityCert {
    pub chunk_id:        u64,
    pub round:           u32,
    pub chunk_hash:      [u8; 32],

    /// Aggregated prevote signature meeting 2/3 stake quorum.
    pub prevote:         AggregatedVote,
    /// Aggregated precommit signature meeting 2/3 stake quorum.
    pub precommit:       AggregatedVote,

    /// Active validator set used for voting. Must match the chunk's
    /// active_validator_set_root.
    pub active_validator_set_root: [u8; 32],
}

pub struct AggregatedVote {
    /// Bitmap of which validators in the active set signed.
    pub aggregation_bits: BitVec,
    /// BLS aggregate G2 signature (compressed).
    pub signature:        [u8; 96],
}

pub struct QuorumCertificate {
    pub data:      FinalityVoteData,
    pub aggregate: AggregatedVote,
}
```

A `FinalityCert` is constructed by the consensus engine / chunk-aggregator
nodes once 2/3 prevote + 2/3 precommit messages for the same
`(chunk_id, round, chunk_hash)` have been seen. The runtime applies rewards and
slashes based on evidence; it does not construct consensus certificates. The
certificate is the input to the recursive proof at the next checkpoint.

---

## 7.10 Checkpoint

A checkpoint is the public-input commitment carried by a recursive proof. It
represents "the chain from genesis through chunk `index`".

```rust
pub struct Checkpoint {
    pub chain_id:                 u64,
    pub index:                    u64,    // 0 = genesis checkpoint
    pub start_height:             u64,
    pub end_height:               u64,
    pub start_block_hash:         [u8; 32],
    pub end_block_hash:           [u8; 32],
    pub start_state_root:         [u8; 32],
    pub end_state_root:           [u8; 32],
    pub end_validator_set_root:   [u8; 32],

    /// Merkle root over every chunk_id covered by this checkpoint, with the
    /// chunk's hash as the leaf. Used for historical inclusion proofs.
    pub history_root:             [u8; 32],

    /// Proof system version at the time of this checkpoint.
    pub proof_system_version:     u32,
}
```

Checkpoints chain by `start_*` matching the previous checkpoint's `end_*`.
For checkpoint index > 0, `start_height` is the previous checkpoint boundary
height, while the first covered chunk begins at `start_height + 1`. Checkpoint
index 0 is the genesis checkpoint.
There is no signature on a checkpoint — its authenticity comes entirely from
the recursive proof that attests to it.

## 7.11 Proof artifacts (opaque to engine schema, defined by ProofSystem)

```rust
pub struct BlockProof {
    pub height:      u64,
    pub block_hash:  [u8; 32],
    pub public_inputs: BlockProofPublicInputs,
    pub proof_bytes: Bytes,   // backend-defined
}

pub struct ChunkProof {
    pub chunk_id:    u64,
    pub chunk_hash:  [u8; 32],
    pub public_inputs: ChunkProofPublicInputs,
    pub proof_bytes: Bytes,
}

pub struct RecursiveCheckpointProof {
    pub checkpoint_index: u64,
    pub checkpoint_hash:  [u8; 32],
    pub public_inputs: RecursiveProofPublicInputs,
    pub proof_bytes: Bytes,
}
```

Public-input schemas are defined in [10-proof-system](10-proof-system.md). The
`proof_bytes` field is treated as opaque outside the `ProofSystem`
implementation. Field schemas are still SCALE-encoded so they survive
codec evolution.

Block proofs are **not** included in blocks — they are gossiped separately on
a dedicated proof topic. Chunk proofs are gossiped per-chunk. Recursive
checkpoint proofs are gossiped per-checkpoint.

## 7.12 Canonical hashing

| Use | Hash | Reason |
|---|---|---|
| Block hash | BLAKE3 | Hot path; ~3× faster than SHA-256 in software |
| Header signature input | BLAKE3 | Same |
| Transactions/votes/slashings roots | BLAKE3 | Computed per block |
| State trie nodes | BLAKE3 (configurable) | Hot path |
| Chunk/checkpoint hash | BLAKE3 | Consistency with block hashing |
| VRF output derivation | SHA-256 | BLS hash-to-curve compatibility |
| Signing-domain hashing | SHA-256 | BLS uniformity / EIP-2333 compat |
| Proof-system public-input hash | Backend-defined | SP1=SHA-256; Plonky3=Poseidon |

We deliberately split fast hashing (BLAKE3) from BLS-compat hashing (SHA-256).
The recursive proof verifier handles both because both occur in public inputs.

## 7.13 Genesis block

```
Header {
    version:           1,
    height:            0,
    slot:              0,
    parent_hash:       [0; 32],
    proposer_index:    0,
    vrf_proof:         [0; 96],
    state_root:        <init_genesis>,
    transactions_root: [0; 32],
    votes_root:        [0; 32],
    slashings_root:    [0; 32],
    validator_ops_root:[0; 32],
    da_root:           [0; 32],
    runtime_extra:     [0; 32],
    gas_used:          0,
    gas_limit:         ChainSpec.genesis_gas_limit,
    timestamp:         ChainSpec.genesis_time,
    signature:         [0; 96],
}
```

Genesis has no signature; nodes accept it because they match its hash against
`ChainSpec.genesis_hash`. Similarly, the genesis *checkpoint* (`index = 0`) has
no recursive proof — clients accept it from `ChainSpec.genesis_checkpoint`.

## 7.14 Wire-shape summary

| Object | Where it appears | Hash | Proof |
|---|---|---|---|
| `Header` | every block | BLAKE3 | covered by `BlockProof` |
| `Body` | every block | indirectly via root fields in `Header` | covered by `BlockProof` |
| `BlockProof` | gossiped on block-proofs topic; not in any block | backend hash | (self) |
| `Chunk` | derived by chunk-aggregator nodes | BLAKE3 | covered by `ChunkProof` |
| `FinalityVote` | in blocks' bodies AND on vote topics | none (signed) | aggregated into `FinalityCert` |
| `FinalityCert` | input to recursive proof; persisted in archive nodes | none | covered by `RecursiveCheckpointProof` |
| `Checkpoint` | public input of `RecursiveCheckpointProof` | BLAKE3 | (self) |
| `ChunkProof` | gossiped on chunk-proofs topic; persisted briefly | backend hash | (self) |
| `RecursiveCheckpointProof` | gossiped on checkpoint topic; persisted (small) | backend hash | (self) |
| `SlashingEvidence` | in blocks' bodies; runtime applies penalty | none | inherited from block |
| `Deposit` / `VoluntaryExit` | in blocks' bodies | none | inherited from block |
